# 小红书文稿 - 长文拆分工具

把一篇长文拆分成多篇小红书文稿（每篇约 600 字），保持风格和故事连贯性，方便后续做图文和封面。

> 适用场景：  
> - 已经有一篇长文（公众号文章、复盘、长故事等），想改造成一组小红书图文  
> - 想把一整套系列内容拆成多篇小红书连载  
> - 想先拆成文稿，再用《小红书排版-图文生成》把每篇做成封面+内容图

---

## 一、完整流程总览

1. 前置检查：确认原文、目录等是否就绪  
2. 第一步：分析长文，识别拆分点，产出《拆分分析》文档  
3. （等待用户确认拆分方案）  
4. 第二步：按拆分方案重组文章，生成多篇小红书文稿  
5. 第三步：如用户需要，再交给《小红书排版-图文生成》生成封面和内容图

对应的典型目录结构：

```bash
小红书/[选题名]/
├── 00_原文路径.md              # 记录原文路径
├── 01_拆分分析.md              # 拆分方案（等待用户确认）
├── 小红书_01_[标题].md        # 第一篇文稿
├── 小红书_02_[标题].md        # 第二篇文稿
├── [标题1]/                    # 第一篇输出（图文生成后）
│   ├── cover.jpg
│   └── content_*.png
└── [标题2]/                    # 第二篇输出
    ├── cover.jpg
    └── content_*.png
```

---

## 二、前置检查

在真正拆分前，必须完成三项检查。

### 1. 原文文件是否存在

- 检查用户提供的长文路径是否正确  
- 能正常读取到正文内容  
- 如果文件不存在或无法读取，要提示用户重新确认路径

### 2. 原文格式是否正确

- 是否有明确标题（一级标题或首行标题）  
- 是否按段落分行，而不是一整块大段文字  
- 是否包含图片引用（如有的话需要保留）  
- 如果格式极度混乱，要先给出建议（例如先用长文排版工具清洗）

### 3. 输出目录检查

- 目标目录：`小红书/[选题名]/`  
- 如果不存在：可以提示用户是否创建  
- 如果已经存在：  
  - 询问是覆盖现有内容，还是使用一个新的选题名目录  
  - 避免直接覆盖用户之前生成的图文

---

## 三、第一步：分析长文，识别拆分点

### 1. 读取长文并理解结构

- 统计全文字数（中文字符数为主）  
- 识别文章结构：标题、二级小标题、段落、图片位置  
- 粗略判断文章属于：故事型 / 观点型 / 教程型 / 复盘型

### 2. 识别拆分逻辑

**核心原则：一篇小红书文稿 = 一个相对完整的故事/观点单元。**

- 按「核心观点 / 关键故事段落」拆分  
- 目标字数：每篇约 600 字（500-800 字都可接受）  
- 确保每篇可以**独立成文**，不是断在半句上  
- 前后篇之间要有自然的逻辑承接

需要特别避免：

- 不要把「太虚」「太空」的部分单独成篇  
  - 例如纯预告、纯感悟、纯鸡汤——可以和具体故事合在一起  
- 每篇都要有**故事感或画面感**  
  - 有人、有事、有场景，读者能脑补出画面

### 3. 生成《拆分分析》文档

输出文件：`01_拆分分析.md`，包括：

- 建议拆分成几篇（总篇数）  
- 每一篇的暂定标题 / 小标题  
- 每篇覆盖的核心内容（2-3 行说明）  
- 每篇预计字数范围  
- 图片大致分布（哪些图放在哪一篇）

示例结构：

```markdown
# 小红书拆分分析 - 选题：[选题名]

## 总体建议
- 原文字数：约 7800 字
- 建议拆分：5 篇
- 每篇字数：500-900 字

## 篇目规划

### 小红书_01_完美牛马的代价
- 核心：为了显卡两个月工资，精确算计后的自愿加班
- 内容：从买显卡开头，讲"完美受害者"的心理机制
- 图片建议：办公室/桌面场景

### 小红书_02_好学生的第一次反叛
- 核心：记过、造假、第一次反叛的内在逻辑
- ...
```

### 4. 等待用户确认

- 把《拆分分析》给用户看  
- 询问用户是否确认按该方案拆分  
- 用户可以：
  - 调整篇数（例如 5 篇改 4 篇）  
  - 调整每篇重点  
  - 标注不想写/不想公开的部分  
- **在用户明确同意前，不要擅自执行第二步拆分。**

---

## 四、第二步：根据拆分点重组文章

当用户确认《拆分分析》后，才进入真正的拆分与重写。

### 1. 路径与命名清理（重要）

标题会进入文件名和目录名，必须先做路径清理：

- 清理非法字符：`/ \ : * ? " < > |`  
- 限制长度：最多 50 个字符，超出部分截断  
- 去除前后空格  

示例：

- 原文件名：`小红书_01_如何用骂街做一款摸鱼软件.md`  
- 清理后标题：`如何用骂街做一款摸鱼软件`  
- 输出目录：`小红书/[选题名]/如何用骂街做一款摸鱼软件/`

### 2. 生成独立 Markdown 文稿

- 文件命名：`小红书_01_[标题].md`、`小红书_02_[标题].md` …  
- 头部可加简单 metadata（选题名、系列信息等）  
- 内容基于原文重组，不是生硬切割：
  - 适当补足承接句  
  - 为每篇补一个有力开头和自然结尾  
  - 避免读者感觉「这好像是从中间截断的一段」

### 3. 格式调整规则

**核心原则：按小红书阅读体验重新排布。**

- 按语义自然分段，一个完整意思/观点为一段  
- 一段建议 20-40 字，最长不超过 50 字  
- **普通段落之间不加空行**（直接换行）  
- 只有**单行加粗段落**（`**文字**`）上下保留空行，用作情绪锤子或小标题  

格式检查清单：

- [ ] 每段字数大致在 20-50 字（避免一大坨或太碎）  
- [ ] 普通段落之间没有多余空行  
- [ ] 单行加粗段落上下有空行  
- [ ] 开头有小标题（一级标题后紧跟一个二级标题）  
- [ ] 小标题 3-5 个为宜，不要过密  
- [ ] 图片用相对路径，且文件存在

### 4. 图片处理

- 保留原文中的所有图片  
- 图片路径使用相对路径（相对于文稿文件）  
- 图片大致放在对应段落附近，保持阅读节奏  
- 如发现图片文件不存在：记录在一份「缺失图片清单」中提示用户

### 5. 开场设计

#### 第一篇开场

优先使用以下两种方式之一：

1. **直接上结果/冲突**  
   - 第一句话就是冲突或结果，不做长铺垫  
   - 然后用一两句自然回到背景  
2. **金句/情感开场**  
   - 用一句有画面感的金句直接把人拉进情绪里  

要避免：

- 啰嗦的自我介绍：「先简单介绍一下我自己…」  
- 太技术性的背景说明，一上来就解释概念

#### 后续篇开场

- 直接进入故事或关键场景  
- 避免「上篇说到…」「接着上篇…」这类报幕式开场  
- 读者即使只看这一篇，也应该能看懂

### 6. 结尾设计

#### 中间篇结尾

- 用省略号制造悬念，不要写「下篇细说」  
- 通过一个未完全展开的矛盾或问题来收尾

#### 最后一篇结尾

- 适度预告下一个系列，但不要废话  
- 可以用一句升维的金句 + 省略号作结

### 7. 生成文件与记录

- 所有文稿统一保存在：`小红书/[选题名]/`  
- 为了可追溯性，生成一个 `00_原文路径.md`，记录：
  - 原文路径  
  - 原始标题  
  - 简要说明（如版本、日期等）

## 五、第三步：与图文排版技能协同

当文稿拆分完成后，可以提示用户是否需要继续生成图文：

- 如果用户说「要生成图片 / 做小红书图文 / 做封面」：  
  - 引导或直接交给《小红书排版-图文生成》  
  - 由该技能完成封面和内容图的生成

推荐的整体链路：

```text
长文 → 小红书文稿-长文拆分 → 用户确认拆分方案 → 生成多篇文稿
    → （可选）小红书排版-图文生成 → 每篇产出封面 + 内容图
```

## 六、与其他能力的关系

- 上游：  
  - 《长文 - 主调度中心》（article）  
  - 任何生成/整理长文的写作流程
- 下游：  
  - 《小红书排版-图文生成》：基于这些文稿生成封面和内容页  
- 互补：  
  - 标题相关技能（如 `title-master` 或本目录下的标题生成）可辅助生成适合封面的标题/副标题

---

## 七、核心原则总结

1. **先分析，后执行**：拆分方案一定要先让用户确认  
2. **一篇一故事**：每篇都有完整的情绪线或信息线  
3. **每篇约 600 字**：500-800 字弹性区间，不必死卡  
4. **保留所有关键信息与图片**，不要因拆分而丢失重要上下文  
5. **路径命名要干净可复用**，为后续图文生成打好基础  
6. **格式适配小红书阅读习惯**，短段落 + 适度加粗 + 强情绪开场  
7. **真诚 > 套路**：宁可少一点技巧，也不要为了「爆」而假
